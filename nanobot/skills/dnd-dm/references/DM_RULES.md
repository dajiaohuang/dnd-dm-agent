# DM 运行规则

数据库保存战役状态与完整 Snapshot；`dnd-engine` 只负责确定性规则计算和真实随机掷骰。

## 1. 战役启动

### 1.1 确认战役

1. 从当前对话继承已经明确的 `campaign_id`。
2. 没有明确战役时，使用 `dnd-campaign-manager` 列出活动战役。
3. 只有一个活动战役时可以选用；有多个时让玩家选择。
4. “切换战役”只改变本对话后续使用的 `campaign_id`，不自动读档。
5. 所有数据库查询和状态变化都必须限定当前 `campaign_id`。

禁止根据相同槽位号、角色名或模组名推断战役身份。

### 1.2 玩家入口

确认战役后提供与当前状态相符的选项：继续当前状态、查看并恢复 Snapshot、创建新战役，
或查看当前队伍、任务和剧情摘要。玩家明确指定战役和槽位时直接执行校验与恢复；只有
信息不足或存在多个候选时才请求选择。

### 1.3 新战役

先确定战役名称、模组来源、规则版本及初始队伍或建卡计划。

**模组来源二选一：**
1. **使用数据库已有模组**——先 `dnd_module action=status` 列出已导入的模组，让用户选。
2. **导入新模组**——用户指定本地路径或上传 PDF/Markdown 文件（从 `[Attachment: <path>]` 提取路径）。

创建独立 `campaign_id`，初始化世界、队伍与剧情摘要。
不得复用另一战役当前状态；以旧战役为模板必须走克隆流程。

**创建新战役后必须立即创建初始 Snapshot（槽位 1，标签 "初始状态"），作为本次开团的
第一个恢复点。后续每个章节节点、升级、关键战斗前后再创建新槽位。**

推荐使用 `dnd_campaign start` 一键完成（创建 + 初始存档 + 可选导入模组）；
CLI 是维护后备：

```
dnd_campaign action=start name="战役名" module_name="模组名" [source_path="<path>"]
```

```powershell
python -m nanobot.dnd.db.cli campaign create --name "战役名" --module "模组名"
python -m nanobot.dnd.db.cli save create --campaign <campaign-id> --label "初始状态" --workspace "<workspace>"
```

## 2. 数据权威与一致性

### 2.1 权威顺序

发生冲突时按以下顺序处理：

1. 当前 `campaign_id` 的数据库状态；
2. 最近成功提交的审计版本；
3. 当前战役的剧情摘要与事件记录；
4. 对话中的临时描述。

展示内容不是权威数据。任何机械变化必须先完成引擎计算，再在一个数据库事务中更新
相关聚合并写入审计。

### 2.2 完整状态

战役当前状态至少覆盖：

- 世界、章节、场景、时间、派系、任务和关键 NPC；
- 队伍、角色、生命值、资源、装备、法术位和持续效果；
- 当前战斗、顺位、轮次、环境和战斗日志；
- 剧情摘要、开放线索、事件与模组场景状态。

禁止只更新展示文本而不更新数据库，也禁止只更新角色而遗漏由同一动作引发的世界、
任务或战斗变化。

### 2.3 并发与版本

同一战役的状态变化必须检查 `state_version`。发现版本落后时重新读取并重新计算，不能
用旧上下文覆盖新状态。一次动作涉及多个聚合时必须同事务提交，失败时整体回滚。

## 3. Snapshot 存档与恢复

存档操作优先使用 `dnd_save` 原生工具（`create`/`list`/`verify`/`restore`/`delete`/`export`），
CLI 为维护后备。`restore` 默认在加载前自动保存当前状态（`auto_save=true`）以防止误操作丢失进度。
Archived 战役拒绝 save/load 操作。

### 3.1 创建 Snapshot

以下时机应创建完整 Snapshot：

- 玩家主动要求保存；
- 章节结束或进入下一章前；
- 升级或长休完成后；
- 关键战斗、不可逆选择或重大资源消耗前；
- DM 判断当前节点适合作为恢复点时。

每次保存创建新槽位，不覆盖旧槽位。标签应简短说明节点。保存成功后报告槽位、章节、
地点和校验值前缀。

### 3.2 Snapshot 内容

完整 Snapshot 必须包含当前战役元数据和所有会随游玩进度变化的状态聚合。角色、世界、
队伍、战斗、剧情摘要、事件和场景进度随存档保存；模组原文、章节元数据和场景索引属于
静态导入内容，不重复嵌套到 Snapshot，恢复时也不得删除或替换。历史 Snapshot、审计、
骰点历史和全局规则库同样不嵌套到 Snapshot 中。

若 `USER.md` 存在当前战役玩家—角色标记区块，保存前先同步到
`Campaign.config.user_md_player_roles`，使其随 Snapshot 保存。

### 3.3 列表与校验

存档列表必须使用 `(campaign_id, slot)` 查询。恢复前依次校验：

1. 数据库记录属于当前战役；
2. Snapshot 内嵌 `campaign_id` 与请求一致；
3. Snapshot 格式和 schema 版本受支持；
4. SHA-256 校验值一致；
5. 所需状态集合完整。

任一校验失败都停止恢复，不尝试改用其他战役或跳过错误。

### 3.4 恢复

恢复必须在单一数据库事务内执行：

1. 捕获恢复前完整状态；
2. 用选定 Snapshot 替换目标战役当前状态；
3. 保留 Snapshot 与历史审计；
4. 写入恢复审计和完整前后状态修订；
5. 提交后刷新 Agent 上下文；
6. 将恢复后的玩家—角色区块投影回 `USER.md` 对应标记区块。

恢复不会切换到另一个战役，也不会改变其他 `campaign_id` 的状态。

### 3.5 撤销

撤销按当前战役审计顺序恢复完整前态，遵守配置的最大撤销次数。撤销读档后同步刷新
数据库上下文与 `USER.md` 战役区块。撤销不删除任何 Snapshot。

## 4. 模组管理

### 4.0 模组入库

战役创建时的 `--module` 仅设置名称标签，不代表模组内容已存在。使用模组事实前必须导入
用户提供的 UTF-8 Markdown 章节目录，并核验章节数与场景数。

**导入前必须先检查模组是否已存在**（`module list` 或 `dnd_module action=index`）。
如果战役已有同名模组且 chapter/scene 数非零，直接使用已有数据，不得重复导入或解析源文件。
仅在以下情况导入：
- 战役没有任何模组，或
- 用户明确要求重新导入（先 delete 再 import）

```powershell
# 先检查
python -m nanobot.dnd.db.cli module list --campaign <campaign-id>
# 确认缺失后再导入
python -m nanobot.dnd.db.cli module import --campaign <campaign-id> `
  --name "<模组名>" --path "<绝对目录>"
```

没有合法本地模组源时必须明确报告缺失，禁止根据模型记忆重建已出版模组。
PDF 使用专用结构解析器；DOCX、PPTX、XLSX、HTML 等附件由 MarkItDown 转成 Markdown，
再按章节、场景和检索块入库；每个检索块使用 BGE-M3 生成 Dense 向量。向量优先写入
ChromaDB（若已配置 `CHROMA_DB_URL` 或 `CHROMA_DB_PATH`），未配置时回退到 SQL
`embedding_json` 列。导入完成后嵌入向量异步写入 ChromaDB，不阻塞 SQL 事务。
频道消息中的本地附件路径只有在玩家明确要求作为模组来源时才可导入。

其中 PDF 使用分页与书签感知的专用解析器：恢复章节/附录/场景/房间标题，重排并合并
排版换行，保留页码范围，再按同一场景内 1200 字上限和约 100 字重叠切块。Chunk 禁止
跨标题边界；目录项不进入 Dense；属性块、房间和列表应带独立类型元数据。

### 4.1 事实来源

只把当前战役绑定的模组内容视为剧情事实。SRD 提供机械规则，不提供特定模组剧情。
模组未说明的细节可以做最小、可逆的氛围填充，但不能改变关键人物、线索、节点、奖励、
地图连接或章节结局。

### 4.2 场景加载

- 只加载当前位置和即将互动所需的场景片段。
- 不提前向玩家泄露后续章节或未发现内容。
- 场景索引必须从当前模组源生成，不能使用其他机器或战役遗留索引。
- 通过 `dnd_module` 的 Dense 检索定位候选块，再展开完整场景；不得只依赖摘要命中。
- 进入新场景时更新数据库场景状态和剧情摘要。

### 4.3 推进约束

推进前确认当前关键节点及完成条件、必要线索、NPC 已知信息、章节转换条件，以及是否
需要先升级或创建 Snapshot。发现偏离时通过模组内线索自然引导，不能强制传送、凭空
生成关键道具或替玩家作决定。

## 5. 玩家行动与检定

### 5.1 玩家先行动

先描述可感知环境，再问玩家做什么。不得替玩家决定动作、对白、路线、目标、法术、装备
或资源消耗。多个角色都可行动时，询问由谁执行。

### 5.2 何时检定

同时满足以下条件才检定：结果存在真实不确定性、成功与失败都有意义、规则允许该行动，
且当前角色有能力尝试。自动成功或不可能完成的行动不掷骰。先确认行动，再确定能力、
技能、优势或劣势和 DC。

### 5.3 检定流程

1. 从数据库读取角色完整状态和环境修正。
2. 用 `dnd-engine` 真实掷骰。
3. 用引擎结算总值、优势或劣势和结果。
4. 说明可公开的骰点、加值来源和结果；不要提前泄露隐藏 DC。
5. 将后果完整写入数据库并记录审计。

禁止心算随机数、为了剧情改骰、事后补设 DC，或在未调用引擎时声称已经掷骰。

## 6. 战斗

### 6.1 开战

确认参战者、位置、先攻、AC、HP、环境和初始状态。先攻必须真实计算。突袭、隐蔽、
掩护、光照和距离按 SRD 与模组场景处理。

### 6.2 回合

严格按先攻顺序执行。每个单位回合跟踪移动、动作、附赠动作、反应、资源、状态和持续
时间。角色仍有可用动作时不要擅自结束其回合。NPC 只使用实际知道的信息，不针对玩家
未公开计划进行全知反制。

### 6.3 攻击与伤害

攻击顺序：确认目标和武器 → 计算攻击加值 → 掷 d20 → 判断命中 → 掷伤害 → 应用抗性、
免疫和临时 HP → 更新状态。重击、豁免和持续效果必须调用对应引擎函数。

### 6.4 战斗结束

结算经验、掉落、消耗、濒死或死亡、任务和世界影响；更新完整数据库状态与剧情摘要。
重大或章节性战斗结束后创建 Snapshot。

## 7. 角色创建与升级

角色创建使用 `CHAR_CREATION.md` 与模板工厂，逐步确认种族、职业、背景、属性、技能、
装备、法术和姓名。最终确认前不写入正式角色状态。

升级必须基于当前等级递增，逐项处理 HP、职业或子职能力、熟练加值、技能与豁免、专长
或属性提升、法术与法术位、装备资源及派生数值。升级后验证一致性，提交数据库，再创建
Snapshot。禁止从头重建角色覆盖玩家已有选择、物品或状态。

## 8. 叙事与信息边界

- 只描述角色能感知的信息。
- 不泄露房间编号、节点 ID、触发器名称、隐藏数值或幕后推理。
- 不提前剧透 NPC 身份、陷阱、伏击、宝物和后续章节。
- 支线必须短、可回归主线并记录状态；不能无限扩张替代模组。
- 失败要产生合理代价，但不能用任意惩罚剥夺玩家选择。
- 不把现实世界私人信息自动改写成剧情素材。

## 9. 上下文与摘要

常驻上下文只保留当前战役摘要、当前场景、队伍关键资源、未决线索和最近对话。规则、
完整角色卡、旧事件和模组章节按需读取。对话过长时把已确认事实写入剧情摘要和事件，
不以模型记忆替代数据库。

### 事件日志

玩家选择、任务变化、场景进入、NPC 状态变化、战斗结果、资源获得或消耗等会影响后续
叙事的已发生事实，必须及时追加到当前战役事件日志：

```powershell
python -m nanobot.dnd.db.cli event create `
  --campaign <campaign-id> `
  --type <事件类型> `
  --content "<简洁、客观、已发生的事实>" `
  --actor-name "<角色名>" `
  --importance <1-5>
```

事件类型使用稳定短名，例如 `scene_entered`、`quest_updated`、`choice_made`、
`combat_resolved`、`npc_updated`、`resource_changed`。同一事实只记录一次；玩家尚未确认的
意图、DM 的幕后计划和模型推测不得写入。事件会随完整 Snapshot 保存和恢复。

## 10. 输出前自检

1. 使用了正确的 `campaign_id`。
2. 没有替玩家决定。
3. 随机结果确实来自引擎。
4. 模组事实和 SRD 规则没有混淆。
5. 没有泄露隐藏信息。
6. 状态变化已完整持久化并审计，影响后续叙事的事实已写入事件日志。
7. 若宣称保存或恢复，数据库操作确实成功。
8. 下一步给玩家清晰、可行动的选择。
