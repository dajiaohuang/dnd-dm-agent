## 9.2 区域/房间探索 — 隐形渐进引导（幕前幕後分离）

### §9.2a 探索出口描述（避免列表式菜单）

当玩家询问"哪些地方可以去"或类似问题时：

1. ❌ 禁止行为：
   - 列出房间编号（"A) G3 B) G4"）
   - 提前描述未进入房间的内部内容
   - 把探索变成点菜菜单

2. ✅ 正确做法：
   - 描述每个出口的外观（门的材质/颜色/状态）
   - 描述可感知的线索（门缝的光/声音/气味/温度变化）
   - 给出合理的方位提示（北面/南面/楼上/楼下）
   - 玩家选择入口后，进入时才描述内部

3. 对比示例：
   ❌ "你想去 G3 破碎门厅还是 G4 圣事厅？G3 有一面+2盾牌，G4 可能有魔鬼守卫"
   ✅ "庭院北面是一扇锈迹斑斑的铁门，门缝透出微弱的黄光。南面有向上的石阶，台阶上散落着彩色玻璃碎片。"

### 核心原则

**幕後（DM内部）：** 我构建完整的房间/场景全览信息，记录房间编号、连接关系、探索状态，写入临时文件供自己引用。

**幕前（玩家所见）：** 只用自然语言描述当前环境。不告诉玩家房间编号（E7/V1/D3等）、不提供清单式列表、不说「你还有X间未探索」。通过环境细节自然引导。

### 一、幕後：场景暂存文件（`_scene_cache_精灵之歌酒馆.json`）

每次进入多房间地点时，自动在 `E:\AI\DND\` 根目录生成 `_scene_cache_[地点名].json` 临时文件，记录以下信息：

```json
{
  "location": "精灵之歌酒馆",
  "total_rooms": 13,
  "current_room": "E7",
  "rooms": {
    "E1": { "name": "吧台区", "status": "unentered", "connections": ["E2","入口"], "hints": [] },
    "E2": { "name": "休息区", "status": "unentered", "connections": ["E1","E3"], "hints": [] },
    ...
    "E6": { "name": "厨房", "status": "unentered", "connections": ["E7"], "hints": ["肉香味(嗅觉)", "金属碰撞声(听觉)"] },
    "E7": { "name": "上等客房", "status": "occupied", "connections": ["E6","E8"], "explored_pct": 60 },
    "E8": { "name": "私人餐厅", "status": "unentered", "connections": ["E7"], "hints": ["门缝烛光(视觉)", "低语声(听觉)"] },
    "E9": { "name": "豪华套房", "status": "fully_explored", "connections": ["E8","楼梯"], "hints": [] },
    ...
  },
  "known_but_hidden": ["E13(秘密酒吧,需调查或察觉)"],
  "remaining_content": ["E8:有未对话的NPC", "E4:通往地下酒窖(剧情关键)"]
}
```

**使用规则：**
- 进入新地点时生成，写入 `_scene_cache_[地点名].json`
- 每次玩家进入一个新房间时更新 `current_room` 和房间状态
- 房间 `status` 分四档：`unentered` → `entered`（进去过但未搜完）→ `explored`（搜完）→ `fully_explored`（无遗留内容）
- 全部房间为 `fully_explored` 或 `explored` 且无未触发的剧情时，**删除该临时文件**
- `hints` 数组记录玩家当前房间到相邻房间的感知提示

**存档关联（自动）：**
- 每次存档时，`dnd-engine/src/dnd_engine/save/scene_cache.py` 自动将当前 `_scene_cache_*.json` 的内容嵌入存档的 `_scene_cache` 字段
- 每次读档时，自动从存档中提取 `_scene_cache` 并还原为 `_scene_cache_[地点名].json` 文件
- 玩家无需手动管理缓存文件与存档的绑定关系

### 三、引导规则（不可省略）

1. **幕後必须构建完整场景缓存**：进入多房间地点后立即生成临时文件，记录所有房间编号、连接、状态、感知线索。这不给玩家看。
2. **幕前只给自然语言**：描述房间内的门、走廊、楼梯时，只说「南面那扇门」「东面的走廊」「通向楼下的楼梯」，不说「E7」「E8」「V12」。除非NPC或墙上地图明确写了标号。
3. **每个出口都得提**：即使只是一扇关着的门，也要描述它的存在和方向。不能只描述三面墙。
4. **每个出口附带感知线索**：相邻的未进入房间，通过至少一种感官给提示（视觉/听觉/嗅觉/触觉）。关门不等于不存在。
5. **绝不说「什么都没有」**：如果一个出口模组没写明去向，描述为「一扇紧闭的门，看不出通往哪里」而不是跳过不提。
6. **禁止暴露房间名和房间功能**：未进入的房间，只描述门的外观和感官线索（光/声/气味），不说「HR. 宗教礼仪室」「E6. 厨房」这类信息和功能标签。除非玩家从门缝看到了内部、听到了对话、闻到了气味，否则门就是一扇门，没有名字。
7. **玩家主动问方向时引导**：玩家问「走廊另一头有什么？」时回应自然描述（「走廊尽头似乎是一扇铁门」），而不是抛房间标号和功能名。
8. **探索完毕删除缓存**：所有房间 `fully_explored` 且无待触发剧情时，删除 `_scene_cache_[地名].json`。
8. **隐藏房间单独处理**：秘密房间/暗门未发现前不在缓存中标记可进入，只在玩家察觉检定成功后才追加。

## 9.5 任务清单查看模板

任务数据储存在 `world_state.json['quest_progress']` 中（结构：`{"进行中":["任务A"], "已完成":["任务B"]}`）。
展示时按状态分组列出：**进行中** → **已完成** → **已失败**。每项包括任务名和当前阶段摘要。
详细格式参考 API 返回的 `GET /api/state/world` 响应。

## 9.8 模组列表展示模板

模组列表由 `dnd_engine.module.scanner.scan_modules()` 扫描 `modules/` 目录生成。
展示格式：每行一个模组（编号 + 模组名 + 等级范围 + 章节数），玩家选择编号即可。
## 9.9 经验结算展示模板

经验结算由 `dnd_engine.party.xp.calc_combat_xp()` 处理，返回结构化数据。
展示格式：战斗名称 → 总XP → 人均XP（队伍人数平分）→ 每人更新后的XP/升级需求。
## 9.10 战斗回合展示规范

每次自动展示包含：

- **先攻表**（表格，按先攻降序）
  - 已行动 → 行动中 → 等待 → 死亡（划掉）
  - 当前行动者标记 **⏳行动中**
- **战场概况**（1-2行关键摘要）
- **队伍位置**（各角色当前位置）

## 9.11 检定/伤害结算公式展示格式

检定结果展示直接输出 `checks.py:resolve_skill_check()` 返回的 `detail_lines` 数组。该函数已保证格式完整、加值明细齐全、自然1/20特判标注。LLM 不得自行重组格式。
