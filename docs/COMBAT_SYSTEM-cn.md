# 战斗系统

本文档说明 **DM 模式** 与 **骰娘模式** 下的战斗行为，包括回合流程、QQ 绑定、托管角色、反应窗口和状态更新。

## 共享战斗核心

DM 模式与骰娘模式使用同一套确定性战斗管线。模式差异只影响叙述权限和 NPC 控制方式，不改变底层机械规则。

共同规则：

- 战斗只读取实体角色卡。AC、HP、调整值、豁免、先攻、施法数值、物品和持续效果必须来自结构化角色数据。
- 开始战斗前需要确认参战者。没有实体角色卡的角色不能加入先攻。
- 系统为所有参战玩家角色、NPC 和怪物投掷先攻。
- 战斗进入回合制模式，并在战役状态中记录轮次、当前回合索引、参战者、先攻和反应可用状态。
- 通常只有在行动结算、反应处理和状态审计都完成后才推进回合。
- 战斗结束会清理战斗回合状态，并返回自由扮演模式。

## 角色卡与有效数值

每次战斗行动都会获得：

- 已知的行动者角色卡；
- 所有参战者的裁剪机械卡；
- 消息中提到目标时的目标角色卡；
- 相关规则、法术、效果、记忆和公开战役事实。

临时战斗效果不会覆盖基础角色卡。装备、Buff、Debuff、法术、自定义物品和其他临时调整会生成实时 `effective` 机械快照。

`effective` 快照可能影响：

- AC 与 HP 相关数值；
- 属性调整值；
- 豁免和技能检定；
- 先攻；
- 攻击和施法数值；
- 优势或劣势；
- 额外骰；
- 专注检定；
- 一次性消耗效果。

## DM 模式战斗

DM 模式是完整 AI 地下城主模式。战斗中使用与骰娘模式相同的机械引擎，但可以加入叙述、扮演 NPC，并表现已建立的剧情上下文。

DM 模式可以：

- 描写战斗场景和行动结果；
- 扮演在场 NPC 和怪物；
- 使用 NPC 的扮演设定、秘密、动机、战斗行为和计划行动；
- 将战斗与既有战役记忆、当前场景事实关联；
- 在 NPC 和怪物回合操作它们；
- 使用确定性投骰和结构化状态变更结算机械动作。

DM 模式仍然必须：

- 从实体角色卡读取目标 AC、HP、豁免和调整值；
- 不编造机械数值；
- 可能触发反应时，先等待反应决定再投骰；
- 记录投骰、状态变更和战斗事件。

## 骰娘模式战斗

骰娘模式用于辅助真实玩家和真实 DM。它不会自行主持剧情，但保留战役框架、角色卡、记忆、效果、物品和战斗工具。

骰娘模式可以：

- 主持先攻和回合顺序；
- 计算检定、豁免、攻击、伤害、治疗和效果；
- 回答角色卡、法术、规则、物品和战斗状态相关的机械问题；
- 在用户明确指令下更新 HP、物品、效果和记忆；
- `@` 当前角色绑定的 QQ 用户；
- 询问 DM 决定 NPC 和怪物的行动或反应。

骰娘模式不得：

- 作为 DM 描写场景或行动结果；
- 扮演 NPC 或怪物，除非该角色被显式设置为托管；
- 默认给玩家战术建议；
- 推进剧情或决定剧情后果；
- 仅因为角色是 NPC 或怪物就把它视为自动控制。

## QQ 绑定

QQ 绑定按战役独立维护。切换当前战役时，会同步切换可操作的角色绑定。

绑定规则：

- 同一个 QQ 在同一战役中可以绑定多个角色。
- 同一角色也会在 `integrations.qq_user_ids` 中镜像多个 QQ 用户 ID。
- 战斗中，如果一个 QQ 控制多个角色，路由优先选择当前回合的那个角色。
- 战斗外，如果一个 QQ 控制多个角色，可以通过消息中出现的角色名选择角色；无法明确选择时，系统不会随意猜测。
- 删除角色会删除该角色的 QQ 绑定。
- 退出骰娘模式时，只删除骰娘模式自动创建的 DM 绑定，不影响普通玩家绑定。

## 骰娘模式中的 NPC 与怪物绑定

骰娘模式会把 NPC 和怪物绑定给 DM QQ，让真实 DM 操作它们。

进入骰娘模式时：

- 如果 `campaign.config.dice_dm_qq_user_id` 已存在，则使用它；
- 否则，如果配置中只有一个 DM QQ，则使用该 QQ；
- 否则，系统会追问谁是 DM；
- 确定 DM QQ 后，战役内所有 NPC 和怪物都会建立骰娘模式管理的 DM 绑定。

退出骰娘模式时：

- 删除骰娘模式管理的 NPC 和怪物 DM 绑定；
- 保留普通玩家绑定。

如果 DM QQ 发生变化：

- 删除旧 DM 的骰娘模式管理绑定；
- 将 NPC 和怪物重新绑定给新 DM QQ。

服务启动时：

- 现有数据库会自动迁移为支持“一个 QQ 操作多个角色”的绑定结构；
- 已处于骰娘模式的战役会重新同步 NPC 和怪物到已配置的 DM QQ。

## 托管角色

骰娘模式中，没有任何角色是默认托管的。NPC 和怪物也不是天然托管。

只有角色卡显式设置以下字段时，角色才被视为托管：

```json
{
  "control": "hosted"
}
```

或：

```json
{
  "control": "auto"
}
```

托管角色行为：

- 如果托管角色是当前战斗行动者，即使战斗扮演文字关闭，也可以为其行动附带简短扮演文字；
- 扮演文字必须基于该角色自己的 `roleplay` 和 `story_role` 字段；
- 内容仅限该行动中的动作、语气、战斗行为或即时反应；
- 不得借此推进剧情、扮演其他 NPC，或决定剧情后果。

## 回合权限

回合权限由当前模式和当前行动者共同决定。

DM 模式：

- 玩家角色在自己的回合行动；
- NPC 和怪物由 DM 操作；
- DM 战斗输出可以包含叙述和扮演。

骰娘模式：

- 玩家角色由绑定的 QQ 用户操作；
- NPC 和怪物由其骰娘模式 DM 绑定对应的 QQ 操作；
- 如果一个 QQ 控制多个角色，当前回合会选中对应角色；
- 状态查询、退出战斗、退出骰娘模式等命令可在允许时打断待处理的战斗设置或反应窗口。

## 反应

可能触发反应的行动会先打开反应窗口，不会提前显示投骰结果。

反应流程：

1. 行动者声明行动。
2. 系统从参战者角色卡和反应特性中识别可反应角色。
3. 系统显示行动声明和需要的投骰公式，但暂不投骰。
4. 系统 `@` 绑定的 QQ 用户并询问是否反应。
5. 在骰娘模式中，NPC 和怪物的反应会交给绑定的 DM QQ，除非该角色显式托管。
6. 显式托管角色可以自动决定是否反应。
7. 所有反应完成后，系统再投骰、结算行动、应用状态变化并推进回合。

## 扮演与建议开关

战斗扮演文字和战斗建议按玩法模式分别配置。

默认值：

- DM 模式：战斗扮演开启，战斗建议开启。
- 骰娘模式：战斗扮演关闭，战斗建议关闭。

命令：

- `/combatroleplayon`
- `/combatroleplayoff`
- `/combatadviceon`
- `/combatadviceoff`

重要边界：

- 骰娘模式的非战斗输出永远是工具型输出；战斗扮演和建议开关不影响非战斗行为。
- 托管角色行动文字只对该托管角色自己的行动绕过战斗扮演开关。
- 建议开关与扮演开关彼此独立。

## 状态更新与审计

战斗更新会持久化并可审计。

系统记录：

- 投骰和公式；
- HP 变化；
- 物品变化；
- 持续效果变化；
- 专注检定；
- 反应决定；
- 回合推进；
- 角色版本变化；
- 战役事件和待抽取记忆。

因此之后询问“上一轮发生了什么”“为什么 AC 变了”“哪个效果被消耗了”时，系统可以从结构化状态回答，而不是只依赖聊天记录。
